# Puglit — BLUEPRINT (the constitution)

This is the source of truth for how Puglit generates a SaaS. The CLI and the web generator both read this. Humans read it to understand the system; the LLM reads it to know the rules it must never break.

---

## 0. The core idea (read this first)

A SaaS is ~80% the same plumbing every time and ~20% unique business logic. Puglit splits these and treats them differently:

- **The Spine (the 80%)** — auth, payments, email, crons, analytics, admin, deploy. **Assembled deterministically** by copying & configuring proven files. Never written by an LLM. *It compiles on the first try, every time.*
- **The Domain (the 20%)** — your entities, schema, prompts, copy, and your product's unique "engine." **Generated by an LLM**, constrained by a typed contract (`domain.config.ts`) so it can't drift.

> Rule #1: **the LLM never touches the Spine.** If a feature is in the Spine, it is configured, not generated.

## 1. The fixed stack (non-negotiable)

| Concern | Choice | Why |
|---|---|---|
| Framework | **Next.js 15** (App Router) | RSC, route handlers, one deploy unit |
| Language | **TypeScript** (strict) | typed seam = fewer LLM hallucinations |
| UI | **Tailwind + Radix/shadcn** | composable, themeable, accessible |
| DB | **PostgreSQL** via raw `pg` (`lib/database.ts` + `ensureSchema`) | no ORM lock-in, full SQL control |
| DB host | **Supabase Postgres** (via the **Supavisor pooler**, IPv4) | see §6 "hard-won lessons" |
| Deploy | **Fly.io** (Docker, ≥2 always-on machines, rolling) | crons run in-process; cheap |
| Payments | **Stripe** (global) + **MercadoPago** (LATAM) | subscriptions + one-time |
| Email | **Resend** (primary) + nodemailer fallback | deliverability > Gmail SMTP |
| AI | **OpenAI / Anthropic** via a provider adapter | swappable, cost-logged |
| Errors | **Sentry** | |
| Auth | cookie JWT (web) + Bearer (mobile), magic-link, Google OAuth | |

The generator does **not** re-decide the stack. The opinion is the value.

## 2. Module catalog

Two tiers. **Core** is always present. **Optional** modules are turned on/off in `domain.config.ts`. Full status in [`modules/README.md`](modules/README.md).

### CORE (always)
- **app-shell** — Next 15 skeleton, Tailwind/Radix UI kit, theming, i18n (es/en), landing + marketing pages.
- **db** — `lib/database.ts` (pg pool tuned for the pooler, `ensureSchema`), migrations dir, typed query helpers.
- **auth** — JWT cookie + Bearer, magic-link/OTP, Google OAuth, forgot/reset, email verification (soft gate).
- **middleware** — per-route rate-limit map, security headers/CSP.
- **analytics** — `page_visits` + `analytics_events` + `/api/track` (persists funnel events — the fix learned the hard way), GA4 + Clarity + optional Meta Pixel hooks.
- **admin** — admin-auth + minimal dashboard.
- **deploy** — Dockerfile, `fly.toml` (auto_stop off, ≥2 machines, `CRON_PRIMARY_MACHINE_ID`), secrets template.
- **observability** — Sentry wiring.

### OPTIONAL (toggle)
- **payments** — Stripe subscriptions + checkout + webhooks + billing; MercadoPago; coupons; plans (`free`…`lifetime`); `getEffectivePlan` gating; `feature-usage` metering.
- **email-lifecycle** — verification, onboarding drip, cart-abandonment, reengagement, weekly summary, **bounce poller (IMAP)**, `email-logger` (tracked sends).
- **content-blog** — blog/"magazine" with **AI author personas**, SEO (metadata, JSON-LD, sitemap), scheduled publishing (cron), social auto-publish. *(The recipe-site "chefs", the news-site "columnists", etc.)*
- **ai-layer** — provider adapter, prompt-service pattern, **context-builder** (layered hyper-personalization / "MCP"), **guardian** (safety/validation), `ai_cost_log`.
- **engine** — the prediction/compute slot: an external microservice called via `services-config` with API key + **fallback** (Swiss Ephemeris → betting model → recipe generator → …). May be deterministic, AI, or external.
- **profiling** — implicit-event capture → latent profile, progressive questions, recommendations.
- **gamification** — streaks, check-ins, testimonials, live counters.
- **growth** — A/B testing, ads-optimizer, growth report, exit-intent, conversion wizards.
- **mobile** — Expo/React Native app sharing the API + RevenueCat IAP.
- **geo** — cities, geoip, tz-lookup, weather.

## 3. The domain seam (`domain.config.ts`)

The **only** thing that changes per project. A typed declaration that drives BOTH the deterministic assembly (which toggles, which plans) AND the LLM generation (entities, prompts, copy). See the file itself + `examples/`. The LLM must conform to this contract; it cannot invent tables or routes outside it.

```
domain.config.ts  ──┬─→ [deterministic] selects modules, writes plans, env, fly.toml, package.json
                    └─→ [LLM] generates: entities+schema, API routes for entities,
                                          domain prompts, marketing copy, the engine impl
```

---

## 4. The generation pipeline (how a project is born)

```
1. INTERVIEW   (INTAKE.md)        → fills domain.config.ts        [LLM: cheap model]
2. ASSEMBLE    (deterministic)    → copy Spine, apply toggles,    [no LLM]
                                     write plans/env/fly.toml
3. SCHEMA      (LLM)              → entities → SQL + types + RLS   [LLM: reasoning model]
4. DOMAIN      (LLM)              → routes, engine, prompts, copy  [LLM: coding model]
5. SECURITY    (SECURITY.md)      → SAST + deps + secrets +        [tools + LLM red-team]
                                     red-team → auto-fix loop
6. VERIFY                          → typecheck + build + tests gate
7. DELIVER                         → push to user's GitHub repo + SECURITY_REPORT.md
```

Steps 2, 6, 7 are mechanical and guarantee the project deploys. Steps 3–5 are where the LLM works, always inside the typed contract and behind the security gate.

## 5. Conventions the generated code MUST follow

These are extracted from a real production codebase. The LLM must honor them.

- **Schema lives in `ensureSchema()`** (idempotent `CREATE TABLE IF NOT EXISTS`). In production `ensureSchema` is skipped — **DDL is delivered as `scripts/sql/*.sql` to run manually** (the app DB role is not the owner). Never assume a column exists in prod because it's in `ensureSchema`.
- **All background work is fire-and-forget**: `void Promise.resolve().then(() => job()).catch(log)` — never block the request on a cron/side-effect. Crons tolerate failure and log.
- **One cron primary**: with ≥2 machines, only `CRON_PRIMARY_MACHINE_ID` runs scheduled jobs (avoid double execution).
- **Rate-limit every sensitive route** via the middleware map — but keep auth/register generous (≥20/min) so fumbling users and shared IPs (mobile CGNAT) aren't locked out.
- **Premium gating** goes through one function (`getEffectivePlan`), never ad-hoc checks.
- **Parameterized queries only** (pg `$1,$2`), never string interpolation.
- **Email always via `email-logger`** (tracked send + bounce awareness), never raw transporter calls.
- **External compute via `services-config`** with API key + a graceful fallback, never hardcoded URLs.
- **Bilingual by default** (es/en) via a language context; user-facing strings are translatable.

## 6. Hard-won production lessons (bake these in)

- **Supabase direct connection is IPv6-only; Fly egresses IPv4.** Always connect through the Supavisor **pooler** (`aws-N-<region>.pooler.supabase.com`, user `dbuser.projectref`, session mode 5432). A direct `db.<ref>.supabase.co` connection will silently die when the platform drops IPv4.
- **`/api/track` must persist funnel events**, not just page views — otherwise you optimize blind.
- **Email deliverability is a conversion lever.** Gmail SMTP → spam. Use Resend + verified domain (SPF/DKIM/DMARC). Prefer magic-link/OTP as a primary path.
- **Don't paywall before value.** Let users feel the product (a free first result) before `/pricing`.
- **Rate limits are a silent funnel killer.** Tune them; count only real attempts.

---

## 7. The rules (Karpathy-style, enforced on the LLM)

1. **Simplicity first** — generate only what `domain.config.ts` declares. No speculative features.
2. **Surgical** — the LLM writes the domain folder + declared seams; it does not edit Spine files.
3. **Think before coding** — verify the contract (entities, toggles, prod constraints) before emitting code.
4. **Goal-driven** — success = it builds, passes the security gate, and deploys. Test the feature, not just compilation.
5. **No hallucinations** — real schemas, real types, cited sources where the domain needs facts; checksums where determinism matters.
